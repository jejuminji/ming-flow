# MingFlow 이미지 생성·부분수정·3D 변환 플로우 차별점

## 1. 현재 플로우

현재 MingFlow에는 로컬 및 API 기반 이미지 생성과 이미지 기반 3D 생성 기능이 거의 완성되어 있다.

```text
로컬 생성: Qwen 프롬프트 → 이미지 생성 → 미리보기
API 생성: 프롬프트 정제 → GPT 이미지 생성 → 미리보기
3D 변환: 생성 이미지 → Tripo 또는 TRELLIS → 3D 미리보기 및 다운로드
```

API 이미지 편집 노드는 이미 `IMAGE`, 선택적 `MASK`, 수정 프롬프트를 입력받을 수 있다. 다만 사용자가 생성된 결과를 보면서 영역을 직접 선택하고, 필요한 부분만 수정한 뒤, 확정된 이미지를 3D 변환으로 넘기는 통합 경험은 아직 부족하다.

따라서 다음 핵심 플로우를 완성하는 것이 필요하다.

```text
아이디어
  → 이미지 생성
  → 원하는 영역 선택
  → 선택 영역만 부분수정
  → 결과 비교 및 반복 수정
  → 최종 이미지 확정
  → 3D용 전처리
  → 3D 모델 생성
  → 텍스처 확인 및 다운로드
```

## 2. 핵심 차별점

### 2.1 좋은 부분을 보존하는 반복 제작

일반적인 이미지 생성 플로우는 결과가 마음에 들지 않으면 프롬프트를 변경해 전체 이미지를 다시 생성한다. 이 과정에서는 이미 만족스러웠던 실루엣, 구도, 색감, 캐릭터 특징까지 함께 바뀔 수 있다.

MingFlow에서는 사용자가 얼굴, 손, 장식, 무기, 로고, 의상 등 문제가 있는 영역만 선택해 수정할 수 있어야 한다. 좋은 부분은 보존하고 문제만 해결하는 방식이 핵심이다.

### 2.2 3D 제작을 목적으로 한 이미지 편집

부분수정 노드는 단순한 인페인팅 기능에 머물러서는 안 된다. 3D 생성 전에 발생하는 문제를 해결하는 제작 단계여야 한다.

주요 활용 사례는 다음과 같다.

- 가려진 부위 제거 또는 복원
- 좌우 비대칭 완화
- 지나치게 복잡한 장식 단순화
- 캐릭터와 배경의 실루엣 분리
- 손, 무기, 소품의 접촉 오류 수정
- 정면성 및 형태 가독성 강화
- 배경, 바닥 그림자, 글자 및 로고 제거
- 3D 변환이 어려운 얇은 구조 보강
- 잘린 신체나 오브젝트 영역 복원

즉, 예쁜 이미지를 생성하는 것뿐 아니라 **3D 모델로 변환하기 좋은 이미지를 완성하는 것**이 MingFlow의 차별점이다.

### 2.3 로컬과 API의 하이브리드 사용

사용자는 로컬 모델과 API 모델의 구현 차이를 크게 의식하지 않고 동일한 흐름으로 사용할 수 있어야 한다.

- 초기 이미지는 로컬 Qwen 또는 GPT Image API로 생성
- 부분수정은 API 편집 모델로 처리
- 향후 로컬 편집 모델도 동일 인터페이스로 추가
- 결과는 항상 ComfyUI `IMAGE` 형식으로 통일
- 동일한 마스크와 수정 프롬프트를 여러 편집 엔진에서 재사용

이 구조를 사용하면 **저비용 로컬 생성 + 필요한 영역만 고품질 API 수정**이라는 하이브리드 플로우를 제공할 수 있다. 사용자가 비용, 속도, 품질의 균형을 직접 선택할 수 있다는 점도 중요한 차별점이다.

### 2.4 수정 과정의 재현성과 자산화

일반적인 웹 이미지 편집기는 최종 결과만 남는 경우가 많다. MingFlow는 ComfyUI 그래프를 활용해 다음 정보를 모두 워크플로우로 보존할 수 있다.

- 원본 이미지
- 사용자가 선택한 마스크
- 수정 프롬프트
- 사용한 로컬 또는 API 엔진
- 편집 설정
- 단계별 수정 결과
- 최종 3D 입력 이미지

이를 통해 같은 캐릭터의 장비만 변경하거나 동일한 형태에서 여러 디자인 변형을 만드는 작업을 반복하고 재현할 수 있다.

## 3. 부분수정 기능이 반드시 제공해야 할 것

현재 API Edit 노드는 이미 선택적 `MASK` 입력을 지원한다. 그러나 단순히 마스크 소켓을 제공하는 것만으로는 충분하지 않다. 사용자가 이미지 위에서 수정 영역을 쉽게 선택하고 결과를 비교할 수 있어야 한다.

### 3.1 필수 입력

- 원본 `IMAGE`
- 수정 프롬프트
- 사용자가 직접 칠한 영역 또는 외부 `MASK`
- 편집 엔진 선택
- 마스크 확장 크기
- 마스크 가장자리 흐림 정도
- 원본 보존 또는 편집 강도
- 생성할 수정 후보 수

### 3.2 필수 출력

- 수정된 `IMAGE`
- 실제 적용된 `MASK`
- 원본과 수정 결과의 비교 이미지
- 선택 영역이 표시된 마스크 미리보기

적용된 마스크를 출력해야 같은 영역에 추가 수정을 하거나 다른 프롬프트와 엔진으로 변형을 만들 수 있다.

### 3.3 필수 UI 기능

- 노드 내부 이미지 미리보기
- 이미지 위에 직접 칠하는 브러시
- 지우개
- 브러시 크기 조절
- 마스크 반전
- 마스크 전체 초기화
- 마스크 확장 및 축소
- 가장자리 페더링
- Before/After 전환
- 수정 결과 채택 또는 재시도

사용자가 별도의 마스크 노드를 찾아 연결하지 않아도 생성 결과를 바로 클릭하고 칠할 수 있어야 제품 경험이 차별화된다.

## 4. 권장 노드 구성

하나의 거대한 노드에 모든 기능을 넣기보다 영역 선택과 실제 편집 실행을 분리하는 것이 적합하다.

### 4.1 MingFlow Region Selector

```text
IMAGE → 이미지 위에 브러시 선택 → MASK
```

이 노드는 API를 호출하지 않는 로컬 UI 노드다. 사용자가 영역을 여러 번 수정해도 비용이 발생하지 않는다.

주요 기능:

- 브러시 마스킹 및 지우기
- 마스크 반전
- 마스크 확장 및 축소
- 가장자리 페더링
- 마스크 미리보기
- 선택 영역 면적 표시

### 4.2 MingFlow Partial Image Edit

```text
IMAGE + MASK + 수정 프롬프트
  → API 또는 로컬 편집
  → edited IMAGE
```

초기 버전에서는 기존 `GPT Image 2 Edit` 기능을 편집 실행기로 재사용할 수 있다. 이후 로컬 이미지 편집 모델을 같은 입력과 출력 규격으로 추가한다.

영역 선택과 실행을 분리하면 다음 장점이 있다.

- 편집 실행 전까지 API 비용이 발생하지 않음
- 동일한 마스크를 여러 프롬프트에 재사용 가능
- 동일한 마스크로 로컬/API 결과 비교 가능
- 자동 선택 노드나 외부 마스크 노드와 조합 가능

## 5. 목표 플로우

```text
로컬 Qwen 또는 GPT 이미지 생성
  ↓
이미지 미리보기
  ↓
MingFlow Region Selector
  ↓
MingFlow Partial Image Edit
  ├─ 결과가 만족스럽지 않음 → 영역 또는 프롬프트 수정 후 재실행
  └─ 결과 확정
       ↓
3D용 이미지 검사
       ↓
배경 제거 및 실루엣 정리
       ↓
Tripo API 또는 TRELLIS 로컬
       ↓
3D 미리보기, 텍스처 추출 및 다운로드
```

## 6. 구현 우선순위

### 6.1 MVP

1. 이미지 위에서 직접 사용하는 브러시 마스크 편집 UI
2. 마스크 확장, 축소 및 페더링
3. 기존 GPT Image Edit 노드와 연결
4. 원본과 결과의 Before/After 미리보기
5. 편집 결과를 Tripo 또는 TRELLIS로 직접 전달
6. API 호출 전에 편집 대상과 호출 횟수를 명확히 표시

### 6.2 후속 기능

- 객체 클릭 기반 자동 선택
- 자연어 기반 자동 마스크 생성
- 여러 수정 후보의 나란히 비교 및 선택
- 수정 이력과 이전 결과 복귀
- 로컬 이미지 편집 모델 지원
- 3D 제작용 자동 보정 프리셋
- 이미지의 3D 변환 적합도 검사
- 가림, 잘림, 복잡한 배경, 얇은 구조 자동 감지
- 감지된 문제에 맞는 수정 프롬프트 추천

## 7. 최종 정의

MingFlow의 차별점은 단순히 로컬 및 API 이미지 생성 기능을 함께 제공하거나, Edit 노드에 마스크 입력을 지원하는 데 있지 않다.

핵심은 다음 한 문장으로 정의할 수 있다.

> 로컬 Qwen 또는 GPT API로 이미지를 만들고, 필요한 부분만 로컬 편집 모델이나 GPT API로 다시 수정한 뒤, 사용자가 확정한 이미지만 Tripo 또는 로컬 TRELLIS를 통해 3D 에셋으로 변환하는 재현 가능한 하이브리드 제작 플로우.

부분수정 노드는 이미지 생성과 3D 변환 사이를 연결하는 핵심 제작 단계이며, MingFlow를 단순 생성 워크플로우가 아닌 실제 에셋 제작 파이프라인으로 만드는 기능이다.
